# 도메인?
 - 영토, 영역, 소유지, 지식, 영향, 활동 영역 등.. 활동 범위나 필드

## 소프트웨어 개발에서의 도메인 
 - 사용자들의 활동이나 관심사와 관련
 - 사용자가 프로그램, 또는 소프트웨어 서비스를 적용하는 주제 영역을 도메인이라고 한다.
 - 즉, 소프트웨어가 도메인을 반영하도록 만들어야 한다.

## 도메인 모델

### 정의
> 소프트웨어는 도메인의 핵심 개념과 요소들을 통합하고, 그 관계를 정확하게 구현해야한다
 - 소프트웨어는 도메인을 모델링해야한다
 - 도메인은 현실 세계의 일부이고, 단순히 코드로 직접 옮길 수 없다
 - 따라서, 도메인의 추상화인 도메인 모델을 만들어야 한다
 - 도메인 모델은 소프트웨어가 해결하려는 특정 문제 영역(도메인)의 핵심지도
 - 도메인에 존재하는 중요하 개념과 이들 사이의 관계, 그리고 규칙을 표현

### 도메인 주도 설계 (DDD)
 - 도메인의 복잡성이 주는 문제를 해결하기 위한 접근 방법
 - 도메인 모델을 개발 과정의 중심에 두는 방법
 - 현업 전문가, 이해 관계자가 모두 참여해서 함께 도메인 모델을 발전해야한다.
 - 도메인이 모델이 설계와 코드까지 이어져야 한다 (모델 주도 설계) 『 분석 - 설계 - 구현 전과정에서 도메인모델 발전이 이어진다 』
 - 팀 안에서 도메인 모델에 기반한 단일 어휘체계를 만들고, 이를 문서/회의/대화/코드까지 일관되게 한다(보편언어)
 - 도메인에 집중하고 코드와 모델을 일치시키고 명확한 언어를 사용한다.

### 구체적 실천 방법
 - 도메인 모델과 보편 언어를 프로젝트에 기록
 - 용어사전.md
 - 도메인모델.md
 - 도메인모델.drawio

### 도메인 모델 만들기 
1) 듣고 배우기
2) 중요한 것 찾기 (개념 식별)
3) 연결고리 찾기 (관계 정의)
4) '것'들을 설명하기 (속성 및 기본 행위 명시)
5) 그려보기 (시각화)
6) 이야기하고 다듬기 (반복)

### 도메인 모델에서 결정하는 구성요소
1) 엔티티
 - 도메인 모델을 만들 때 사용하는 패턴
 - 도메인 안에 있는 대상이나 개념
 - 고유한 식별자(identity)를 가지고 이를 통해서 개별적으로 구분
 - 생명주기를 가지고 시간의 흐름에 따라 변경될 수 있다.
 - 또한 외부에 대해서 행위와 정책(규칙)을 가진다.

### 도메인 모텔 패턴
 - 도메인/비즈니스 로직을 구성하는 아키텍처 패턴의 한 가지
 - 도메인 모델의 속성과 행위를 모두 포함하는 "도메인 오브젝트 모델"
 - 오브젝트 모델이므로 <복잡한 연관관계, 커스텀 속성, 상속> 등을 사용
 - 트랜잭션 스크립트는 하나의 업무절차(TX)를 처리하기 위한 스크립트(메서드)를 만들고 비즈니스 로직을 순서대로 코드로 작성

### Value Object
 - 도메인 모델에서 식별자가 필요하지 않고 속성/값으로만 구별되는 오브젝트
 - 엔티티가 너무 많은 책임을 가지는 것을 방지하고 특정 속성 관련 행위를 분리해서 엔티티를 더 집중된 상태로 유지하게 한다
 - 원시 타입보다는 도멩니 개념을 더 명시적으로 나타내서 모델의 명확성을 높인다
 - 생성 이후에 상태가 변하지 않고 변경이 필요하면 새로운 객체로 교체
 - 풍부한 기능을 가진다.
 - 자체 유효성 검사도 가능하다

### 원시 타입을 객체로 교체 - 리팩터링
 - 원시 타입의 남용을 개선하고 코드의 명시성과 안정성을 높이는 리팩터링 방법
 - 값 객체는 도메인 모델의 핵심 구성 요소로 값 그 자체의 의미를 담아내고 불변성, 동등성 등의 도메인 로직을 포함하여 설계하는 방식

### 도메인 모델 패턴
 - 단순 도메인 모델 : DB 설계와 유사, 테이블에 대해 하나의 도메인 오브젝트
 - 풍성한 도메인 모델 : 상속, 전략, GoF 디자인 패턴, 연관 관계 복잡한 로직에 적합하지만 DB매핑이 어려움

### 도메인 로직의 API 개발
 - API 개념부터 : application programming interface 
   - '어떤 모듈이 그 외부에다가 이 모듈 내부에 어떤 기능을 사용하려면 이런 인터페이스 이런 메소드를 사용해서 접근해야한다' 이런식으로 공개하고 있는 인터페이스를 총칭해서 API라고한다.
 - 도메인 모델 패턴은 트랜잭션 스크립트 처럼 작업 단위의 절차형 API를 만들기가 어렵다.
 - 도메인 로직의 명확한 작업 단위 API를 제공하는 애플리케이션 서비스가 필요
 - 즉, 도메인에 대한 FACADE를 앞에다가 애플리케이션 서비스로 만들어 도메인 모델 패턴이 구현될 수 있도록 '관문'역할이 필요
   - 이를 통해 외부 레이어에 도메인에 대한 노출성을 줄일 수 있음 
   
※ 정리하면
> 도메인 모델 패턴을 써서 도메인을 잘 설계하고 <br>
> 시스템에 요구되어지는 어떤 기능들을 정의한 작업 단위의 API형태로 어플리케이션 서비스에서 제공한다.
