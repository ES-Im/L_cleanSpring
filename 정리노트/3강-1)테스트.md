# Spring container를 사용하지 않고 테스트
 - 테스트는 구현에 대해서 하지 말고 "인터페이스에 대해" 하는 것이 기본 원칙
   - 이유 : MemberService를 테스트한다고 치자 -> 미래에 Register관련 기능이 확장이 된다면 MemberService를 분할해야할 시점이 온다 -> 그러면 Test도 분할을 해야함
   - 물론 검증해야할 것은 내부기능이고, 이가 리팩토링 되면 테스트도 바뀌는게 당연하지만, 기능이 크게 바뀌지 않았는데 내부 구조를 리팩토링했더니 테스트도 계속 리팩터링이 필요하다 싶으면 테스트 대상 설정을 잘못 한거라고 생각해야한다.
   - 그래서 MemberService 자체가 아닌, MemberRegister이라는 '인터페이스'를 테스트 하는게 먼저

## Mock(Mockito)과 Stub을 이용
 - Mock : 
   - 단지 테스트가 동작하게 하기 위해서 기본적으로 호출이 됐을 때라도 나잊 않고 적절한 응답값 미리 지정한 걸 돌려 주는 것
   - 실제 이 오브젝트하고 테스트가 수행되는 동안 어떤 인터랙션이 있었는가까지 검증하는 기능 제공
   - Mock 테스트를 쉽게 할 수 있게 만든 라이브러리가 Mockito
 - Stub : 
   - 진짜 객체 대신 끼워 넣는 단순한 가짜 구현체
   - 검증은 관심없고, 테스트가 진행되게만 해줌
   - 구현 방식: 직접 클래스/익명 클래스/람다로 만들기 쉬움
   - DB 없이 ID를 세팅하려고 ReflectionTestUtils 같은 걸 함께 쓰는 경우가 흔함
 - Mock과 Stub 예시 : `MemberRegisterManualTest.java`

## 인터페이스가 있는 통합테스트를 하는데 구현체가 없다면?
 - `@TestConfiguration` 이용 
   - 예시 : `MemberRegisterTest.java`
   - `@Import` : TestConfiguration을 Inner class로 선언하면 알아 구현체를 인식하지만, 외부 클래스로 선언했다면 해당 클래스를 이용하여 적용
 - 나중에 인터페이스를 구현했다면 어떻게 될까?
   - 직접확인해보면, application의 구현체가 아니라 TestConfiguration이 우선된다. 그래서 주석처리를 하거나 지워야함.

# tip : 테스트를 record로 만들어도 된다
 - Autowired가 길어지면 테스트코드 또한 가독성이 낮아짐
 - 이를 이용한 예시가 `MemberRegisterTest.java`
 - 근데 그냥 사용하면 'No ParameterResolver registered for parameter'가 터져버림 
 - 위 resolver를 해결하는 방법은 두개 
   - 방법1. 클래스에 @TestConstructor(autowireMode = TestConstructor.AutowireMode.ALL)
   - 방법2. junit-platform.properties 에 위 테스트컨스트럭쳐를 삽입
     - 테스트컨스트럭쳐같은 경우는 record 형식 테스트를 하다보면 다수 record에서 필요하게 되니 그냥 환경변수와 같이 설정하는 것
     - test/resource/junit-platform.properties 파일에 <br> `spring.test.constructor.autowire.mode=all` 입력 
 - record 테스트에서 @Transactional을 걸면 `Classes annotated with '@Transactional' could be implicitly subclassed and must not be final`<br>이라는 경고창이 뜨는데, 이는 잘못된 경고 무시해도된다. ("@Transactional이 서비스 코드에 붙었을 때 적용할 원리를 테스트에 사용한 @Trasactional에 기계적으로 적용해서 나온 것입니다.")
