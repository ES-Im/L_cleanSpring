# 아키텍처
 - 시스템의 기본적인 구조를 정의
 - 시스템의 중요한 품질 속성에 큰 영향을 미친다.
 - 설계 결정의 기반이 되는 핵심적인 개념
 - 기본 구성 요소와 상호 관계, 제약조건, 원칙 등을 포함하여 이를 이해하는게 중요
---
## 계층형(Layed) 아키텍처
 - 서브 시스템을 계층으로 구조화 하는 아키텍처 스타일
 - 계층은 사용 관계로 연결된다
   - 사용 관계는 일반적으로 단방향이어야 한다 : **상위 계층이 하위 계층의 서비스를 사용하는 하항식 흐름**
   - 각 계층이 하위 계층의 내부 작동 방식을 알지 못하고 **제한된 인터페이스만 사용**하도록 한다(계층 격리, 캡슐화)
   - 어떤 레이어의 변경이 다른 레이어의 컴포넌트에게 가능한 영향을 주지 않도록 해야한다.
   
## 3계층 아키텍처
 3계층 아키텍처는 각 계층의 이름이 그 상황에 어떤 컨셉으로 개발하느냐에 따라 다양함.
 - request → UI(유저 인터페이스) → Domain → Data(DB interface) →
 - request → Presentation → Business Logic → Persistence →
```
 - request → Presentation → Application Service → Infra
                          → Domain(domain model pattern)
                          
 - [request → UI → Domain → Infra] 라는 틀은 같음, 
 - 다른점은
 - 시스템에 요구되어지는 어떤 기능들을 정의한 작업 단위의 API형태로 인터페이스가 도메인의 관문역할을 나누어 'Application Service'가 도메인 앞단에 위치하고 
 - 기술적인 코드가 도메인 레이어 안쪽으로 주입되지 않기 위해, Domain은 순수하게 유지하고
   Application Service가 ‘포트(인터페이스)’를 통해 Infra를 호출
```
 - 완화된 계층형 : 상위 레이어가 바로 아래 레이어만 거쳐야 한다는 엄격 규칙을 완화<br>
   (예: Presentation → Application → Infrastructure 처럼 중간을 건너뛰는 걸 허용하는 형태)

## 헥사고날 아키텍처
 - 계층형 아키텍처의 "단방형 비대칭 구조"가 아닌, 대칭형 아키텍처
   - 대칭형 : 한쪽으로 쭉 이어진 방향성(위 아래, 좌 우)이 아니라 구조차체를 어플리케이션의 내부와 외부 세계라는 대칭 구조를 가진다. 
   - 육각형으로 설명(6 이라는데에는 의미는 없다.)

### 헥사곤의 내부
 - 쉽게 변하지 않는, 중요한 도메인 로직을 담은 "코어 어플리케이션"
   - 도메인 로직을 가진 트랜잭션 스크립트
   - 애플리케이션 서비스와 도메인 모델 패턴을 따라서 만든 도메인

### 헥사곤의 외부
 - 헥사곤과 상호작용하는 모든 것 (Actor)
   - 사용자, 브라우저, CLI 명령, 기계, 다른 시스템
   - 운영환경, DB, 메시징 시스템, 메일 시스템, 원격 시스템
   - 테스트

### 헥사고날 아키텍처의 특징과 혜택
 - 테스트! 운영 시스템에 연결되지 않는 애플리케이션 테스트
 - 애플리케이션과 상호작용하는 액터가 바뀌어도 다시 빌드하지 않고 테스트
 - UI 디테일이나 기술 정보가 도메인 로직 안으로 노출되지 않도록 보호한다(반대도 마찬가지)
 - 컴포넌트를 각각 개발하고 연결하는 방식으로 큰 시스템을 분리
 - 시간이 지나면서 외부 연결으 ㄹ다른 것으로 변경할 수 있다.
 - 기술 요소를 제거했기 때문에 도메인 설계에 집중할 수 있다.

### 헥사고날 내·외부
1. 육각형 자체를 부르는 명칭

| 헥사곤 | 헥사곤, 애플리케이션, 앱, 코어시스템, SuD, SuT |
|-----|---------------------------------|

* SuD: System under Development
* SuT: System under Test

2. 포트와 액터, 어댑터
 - 포트 : 
   - 애플리케이션이 외부 세계와 의도를 가지고 상호작용 하는 아이디어를 캡처한 것
   - 단순히 데이터를 주고받는 것이 아니라, 명확한 목적과 방향을 가지고 외부와 연결
   - **애플리케이션**이ㅣ 정의한 인터페이스로 만들어진다.

 - Adapter : 
   - 애플리케이션의 포트를 액터가 직접 연결할 수 없다면 인터페이스의 변환을 위한 어댑터 도입
   - 브라우저를 통해서 애플리케이션의 회원 가입 포트의 기능 제공 인터페이스를 사용하려면?
     - Provided Interface를 사용하는 Web Controller Adapter를 만든다.
   - 애플리케이션이 가진 회원 정보 저장 포트의 기능 요구 인터페이스로 DB와 직접 연결할 수 없다면?
     - Required Interface를 사용하는 Repository Adapter를 만든다.
   - 다르게 말하면 Application에서 Actor와 소통이 가능하다면 Adapter를 만들지 않아도 된다. 예 - Spring data JPA

* 헥사고날 아키텍쳐를 "포트와 어댑터 아키텍처"라고도 불린다.

3. 명칭 정리

| 종류                                | 기능                                              | 예시          | 연결된 Actor                          | 연결된 Adapter                           |
|-----------------------------------|-------------------------------------------------|-------------|------------------------------------|---------------------------------------|
| Lollipop<br/>(Provided Interface) | 애플리케이션이 가지고 잇는 기능을<br/> 외부의 액터가 사용할 수 있도록 노출한 것 | 회원 등록       | Primary Actor <br/> Driving Actor  | Primary Adapter<br/>Driving Adapter   |
| Socket<br/>(Required Interface)   | 애플리케이션이 동작하는데 필요한 기능을<br/> 외부에서 제공받기 위한 인터페이스   | 메일 발송, mock | Secondary Actor <br/> Driven Actor | Secondary Adapter<br/>Driving Adapter |

### 헥사고날 아키텍처의 오해
1. 어플리케이션 내부에 도메인 계층을 만들어야 한다
    - 헥사고날 아키텍처는 애플리케이션 내부 구현에 대한 원칙이나 요구사항이 없다
    - 도메인 계층을 포함하는 아키텍처는 클린 아키텍처이다. 
    - 헥사고날 아키텍처는 클린아키텍처, 어니언 아키텍처와 다르다.
2. 헥사고날 아키텍처 패키지 구조를 따라야 한다
    - 헥사고날 아키텍처가 요구하는 패키지 구조는 없다
    - 애플리케이션과 어댑터 패키지를 분리하는 것은 바람직하다
    - 포트를 구분된 패키지에 두는 것을 권장한다.
3. 포트는 UseCase라는 접미사를 사용한다 (MemberRegisterUseCase)
    - 포트의 의도를 담은 이름을 사용하기만 하면 된다.
    - For ~ ing 스타일의 권장 네이밍이 있지만, 이를 따를 필요는 없다
4. 애플리케이션에는 도메인 모델만 넣고 JPA 엔티티 등은 어댑터에 둬야 한다
    - 애플리케이션 코드와 포트 인터페이스가 외부 기술에 의존하지 않으면 된다.
   
### 헥사고날 아키텍처가 요구하는 것
1. 애플리케이션은 모든 외부와의 상호작용을 위해서 provided interface와 required interface를 정의한다.
2. 애플리케이션과 상호작용 하는 액터는 런타임에 구성되어야 한다.
3. 애플리케이션은 액터에 대한 코드 의존성을 가지면 안된다.
4. 액터는 정의된 포트를 통해서만 연결해야 한다.
5. 포트 인터페이스는 기술 의존성을 가지지 않는다.

### 헥사고날과 도메인 모델 패턴을 적용한 대칭형 계층 구조 주의점
1. 코드의존성은 외부 → 내부(단, 사용의 흐름은 비대칭적)
2. | 외부 | Adapter → |헥사곤| Application → Domain | |

### 포트 - 인터페이스 네이밍 규칙
- 자바에서 인터페이스 이름을 정의하는건 두가지 
1. 명사 : 실제로 구현하는 클래스의 핵심적인 역할 
   - List, Set, Map, Executor
2. 형용사 : 어떤 클래스에 특징을 부여하기 위한 인터페이스
   - serializable, comparable, iterable

※ 포트를 구성하고 상단에 해당 포트가 어떤역할을 하는지 코멘트를 달아주는 것이 좋다. (API니까)

### 기능은 port인데.. 도메인에서 의존할 경우?
- 예 : 현 프로젝트의 PasswordEncoder
- 해당 인터페이스를 application으로 올려버리면 Member 도메인의 의존방향이 역으로 꺾인다.
- 그러므로 도메인 모델을 표현하는, 도메인 서비스 인터페이스같은 경우는 도메인 레이어 안에 두는 것이 맞다.
- domain과 application을 독립적인 패키지로 구분하긴 했지만, 사실 같은 육각형 내부에 있으니 기본 구형 방식을 깨트리지도 않는다.

### 엔티티 식별자 
 - 고유성 
 - 불변성 : 생명주기 동안 절대 변경되면 안된다. 
   - 참조 정합성을 훼손하고, 연관관계를 깨뜨린다
 - 비즈니스적인 의미로부터 디커플링 되는 것이 낫다
   - 비즈니스적인 의미가 있다면 @NaturalId를 사용
   - Natural key 대신 인조 키(대리 키) 사용 

 ※ @NaturalCache : @NaturalId도 @Id처럼 영속컨텍스트에 캐시로 저장해 두겠다는 것. 네츄럴키로 검색이 잦다 싶으면 쓰면 성능에 유리
