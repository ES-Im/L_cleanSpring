# API 테스트?
- 대상 테스트 : MemberApiWebTest.java

1. @WebMvcTest : Mock MVC 테스트가 가능하게 해준다
   - 특정 api만 실행되도록 함
2. @MockMvcTest
3. @WebMvcTest는 @Controller, @ControllerAdvice, @JsonComponent, Converter/GenericConverter, Filter, WebMvcConfigurer, HandlerMethodArgumentResolver 등 웹 계층과 관련된 빈만 등록한다. 따라서 @Service, @Component, @Repository 등은 등록되지 않는다.
   - @WebMvcTest는 웹 계층을 테스트하기 위한 어노테이션으로, 컨트롤러와 관련된 빈만 등록한다. 따라서 서비스나 리포지토리 등 다른 계층의 빈은 등록되지 않는다.
   - @MockMvcTest는 MockMvc를 사용하여 웹 계층을 테스트하기 위한 어노테이션으로, 모든 빈이 등록된다. 따라서 서비스나 리포지토리 등 다른 계층의 빈도 등록된다.

## `@MockMvcTest` vs `@MockMvcTester`

| 항목            | MockMvc                           | MockMvcTester                     |
| ------------- | --------------------------------- | --------------------------------- |
| 기본 성격         | 전통적인 Spring MVC 테스트 API           | MockMvc를 감싼 AssertJ 기반 fluent API |
| 검증 스타일        | `andExpect(...)` 중심, Hamcrest 스타일 | `assertThat(...)` 중심, AssertJ 스타일 |
| static import | 많이 필요함                            | 거의 필요 없음                          |
| 예외 처리         | 미해결 예외가 바로 던져질 수 있음               | 결과 객체에 실패 정보가 담겨 일관되게 검증 가능       |
| 비동기 처리        | 별도 async 처리 코드가 필요한 경우가 많음        | 기본적으로 비동기/동기 결과를 일관되게 다룸          |
| 체감 난이도        | 익숙하면 강력하지만 문법이 다소 장황              | 더 읽기 쉽고 API가 매끈함                  |

- MockMvc는 .perform을 사용해서 웹 API를 체이닝 형식으로 테스트한 반면, MockMvcTester는 메서드를 Fluent 하게 제공하면서 Assert랑 같이 사용해서 가독성이 좋음
- MockMvc 예시
```java
mockMvc.perform(get("/members/1"))
       .andExpect(status().isOk())
       .andExpect(content().string("ok"));
```
- MockMvcTester 예시
```java
assertThat(mockMvcTester.get().uri("/members/1"))
        .hasStatusOk()
        .bodyText().isEqualTo("ok");
```

## 왜 record 형식이 아니라 @Autowired 했냐?
레코드 형식은 목킹을 못함
```java
@Import(SplearnMemberTestConfiguration.class)
record MemberFinderTest(MemberFinder memberFinder, MemberRegister memberRegister, EntityManager entityManager) {}
```
- 그래서 Repository등 목킹이 필요한 경우는 @Autowired로 필드 주입을 해야함. 그래서 record가 아니라 일반 class로 진행.
```java
@WebMvcTest(MemberApi.class)
class MemberApiTest {
    @Autowired MockMvcTester mvcTester;
    @MockitoBean private MemberRepository memberRepository;
}
```

## application 전체 띄우고 테스트하는 방식
- 대상 테스트 : `MemberApiTest.java`
- 단순한 테스트는 괜찮지만, 복잡해지면 `@WebMvcTest`테스트 또한 복잡해짐
- 단위테스트가 실제 환경과 다른 결과도 나올 수 있기도 해서 API테스트는 `@SpringBootTest` 기반으로 하는걸 권장
- 스프링 지원 기능을 이용해서 하기때문에 서버 전체를 띄우지 않고 빠르게 테스트가능하며, + 캐싱 기능이 있어서 스프링 컨텍스트 자체도 공유를 함
```java
@SpringBootTest         // 전체 스프링 컨텍스트를 띄움
@AutoConfigureMockMvc   // MockMvc를 자동으로 설정하여 주입받을 수 있게 함
// 즉, SpringBootTest만 붙인다고 MockMvc를 사용할 수 있는게 아님, MockMvc를 사용하려면 @AutoConfigureMockMvc도 함께 붙여야 함
@Transactional      // 테스트가 끝난 후 데이터베이스 상태를 롤백하여 테스트 간 데이터 간섭 방지, 예외상황 2가지 하단 참조
public class MemberApiTest {
}
```
### @Transactional 예외 상황
> 1. REQUIRES_NEW : 기존 트랜잭션(작업 1)을 잠시 멈추고 새로운 트랜잭션(작업 2)을 따로 시작
>   - 이 경우에는 DB 작업 1 → 테스트 트랜잭션 소속 → 롤백됨 || DB 작업 2 → 새 트랜잭션 소속 → 이미 커밋됨 → 안 지워짐
> 2. 다른 스레드에서 작업해서 별도 트랜잭션이 되는 경우
>   ```aiignore
>    [스레드 A] 테스트 시작
>    └─ 테스트 트랜잭션 시작
>    └─ 서비스 호출
>        └─ 일부 DB 작업
>        └─ 비동기/백그라운드 작업 시작 → [스레드 B]
>        └─ 별도 트랜잭션 시작 또는 독립 DB 작업
>        └─ 커밋
>    [스레드 A] 테스트 종료
>    └─ 테스트 트랜잭션 롤백
>    ```
