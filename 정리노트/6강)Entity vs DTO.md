# 애플리케이션 포트의 리턴 타입은? Entity vs DTO

## 일단 결론부터
> 애플리케이션 기능 제공 인터페이스의 리턴 타입은 Entity와 Aggregate, 혹은 그 이상의 연관관계로 애플리케이션 로직 수행 결과를 리턴하기 어려운 경우에만 DTO 클래스/레코드를 만들어 사용한다.

## DTO로 꼭 리턴해야 한다는 주장과 반박

> 주장1.<br> 계층형 아키텍처에서 각 계층은 자기 바로 하위 계층에만 의존해야한다는 제약

▶ 현대적 엔터프라이즈 애플리케이션은 완화된 계층형 아키텍처를 주로 사용한다.
  - 의존성 방향이 거꾸로만 안가면 된다.
     
> 주장2.<br> 도메인 계층에 존재하는 엔티티는 애플리케이션 계층에서만 사용하고, 그 밖으로 전달할 떄는 필요한 값을 뽑아서 DTO로 만들어서 전달해서 아키텍처 제약을 따른다

▶ 계층형 아키텍처의 목적은 시스템의 관심사를 분리(separation of concerns) 하는 것
   
> 주장3.<br> 
> 도메인 로직을 가지고 있는 엔티티가 프레젠테이션 로직을 가진 어댑터로 유출되면 <br> 도메인 로직을 담은 메서드를 실행할 수 있는 문제가 있다.
    
▶ 단순조회로직이 아닌, 변경이 있어날 수 있는 도메인 로직을 presentation layer에서 쓰면 안되는 것은 맞다
<br> 하지만, 이러한 유출은 Entity에서만 발생하는 것도 아니고 DTO를 가지고도 Controller에서 "도메인 로직"을 구현할 수 있다.
<br> 즉, DTO로 도메인 로직 수행을 근본적으로 막는다는 건 기술적으로 불가능 한 것.
<br> 그리고 어쩌피 Secondary Adapter에서도 엔티티가 전달되거나 생성되어 리턴된다.
<br> 그러니 Repository Adapter에서 도메인 Entity를 제거할 방법은 없다.
<br> ▶ "그럼 도메인 로직을 실행할 수 있는걸 방지하려면 어떻게 해야하지?"에 대한 대답은 다른 방식으로 해결해야하는 것

> 주장4. <br>
> OSIV 와 결합하면 컨트롤러에서 DB 변경이 일어날 수 있다 <br><br>
> ※ OSIV(Open Session In View) : 트랜잭션이 끝난 후에도 영속성 컨텍스트가 살아있어서 지연로딩이 가능한 패턴

▶ 이건 기술적으로 사실이 아니다, OSIV가 적용됬더라도 트랜잭션 경계는 application(service) 계층이므로, <br> 
application(service) 계층에서 DB 변경이 일어날 수는 있지만, presentation(controller) 계층에서는 일어날 수 없다.

> 주장5. <br>
> 헥사고날 아키텍처(클린 아키텍처, 계층형 아키텍처)에서 application 계층 밖으로 보낼때는 반드시 DTO에 담아서 전달해야한다

▶ DTO를 리턴하는 방식의 가장 큰 문제는 "프레젠테이션 로직이 애플리케이션 레이어로 침투하는 것"
    - View 로직에 따라 엔티티에서 복제되는 DTO 구성이 달라지는데 이를 매핑하는 코드는 모두 어플리케이션 계층까지 영향한다.
    - 즉, DTO를 통해 애플리케이션 계층이 어댑터 계층의 로직에 의존하게 된다.
    - 사실상 상위(adapter) 계층의 로직을 하위(application) 계층이 의존하게 되어 헥사고날 아키텍처/계층형 아키텍처 근간을 꺠는 행위가 된다.
    - 엔티티(애그리거트 루트)를 돌려주고 어떤 정보를 사용자에게 전달할지 선택하는 뷰 로직은 프레젠테이션 관심사를 가진 웹 API 어댑터가 담당한다.
    - 예시로 Application에 엔티티를 반환한다 하면, 프론트에서 "Http response 정보 더 추가해주세요" 라고 하면 application 쪽 고칠거 없이 controller만 고치면 된다.
    - 근데 예외! 엔티티 단위가 아닌 복잡한 << 리포트성 조회 >> 를 할 때는 DTO를 리턴하는데, 이 땐, 도메인 로직 자체가 새로운 정보를 생성하는 거기 때문에 이 자체를 도메인 로직으로 볼 수 있다.

### 정리하면, 
#### Aggregate root 엔티티 자체를 리턴하는 것은 문제가 되지 않고, 리포트성 조회는 새로운 도메인 로직으로서 DTO 형식으로 리턴한다.
#### 개발 가이드와 코드 리뷰를 통해 내부의 로직이 외부에 새어나가는 코드를 작성하지 않도록 주의하자.

#### 주장3과 관련된 엔티티 변경 로직 방지하는 법
 - 정적 분석 도구나 아키텍처 테스트 도움을 받아서 변경 로직이 있는지 테스트 하는게 있다.

### 그리고 OSIV는 의외로 secondary adapter과 application을 결합되는 것을 막아주는 좋은 도구이고 <br> Lazy loading을 적극적이게 활용할 수 있다.
 - (서비스가 아아주커져서 트래픽이 장난아니게 된다면 OSIV를 끄는것도 고려해야함)
