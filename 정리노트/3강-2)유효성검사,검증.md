### 요약 : record 내에서 자바 표준 validation annotation 쓰면 된다.  

# 표준 유효성 검사 로직은 어디에 있어야 하나?
 - 도메인 규칙을 엄격하게 따지면<br>
   이메일 아이디 최소 길이, 이름 최소 길이, 공백 허용 여부 등 검증 항목이 많아질 수 있다.

## 어디서?
1. application Layer에서
    - Application layer가 도메인 계층 진입점이므로, 검증을 두는 것 자체는 가능하다.
    - 다만 이 계층은 “추상적으로 어떤 작업(유스케이스)이 일어나는지”가 한눈에 보이는 가독성이 중요하다.
    - 그래서 검증 로직이 많아지면 Extract Method 등으로 분리하더라도, 흐름이 흐려질 수 있다(가독성 저하 위험).

2. Request DTO에서
   1. record의 compact constructor을 이용(컴포넌트 값을 기반으로 조건을 거는 형태)를 이용해 유효성 검사를 넣을 수 있다.
   2. ★ __JSR-303 Bean validation을 이용(자바 표준 밸리데이션)__
      - 검증과 관련된 기능을 애노테이션을 이용해서 선언적으로 쉽게 처리할 수 있도록 만들어진 자바 표준
      - dependencies를 추가해줘야지 사용할 수 있음
      - `implementation("org.springframework.boot:spring-boot-starter-validation")`

```java
// 1. record 에서 jakarta.validation.constraints. 라이브러리 이용, 유효성 검사 조건 달기
public record MemberRegisterRequest(
        @Email String email,
        @Size(min = 5, max = 20) String nickname,
        @Size(min = 8, max = 100) String passwordHash
) {}

// 2. 파라미터에 유효성 검사대상이라고 표시
public interface MemberRegister {
    Member register(@Valid MemberRegisterRequest memberRegisterRequest);
}

// 3. validation 검사하도록 활성화 (이외 코드 생략)
@Validated
public class MemberService implements MemberRegister {
    @Override
    public Member register(MemberRegisterRequest memberRegisterRequest) {   // 여기서 @Valid 생략 가능  
        // → 이 DTO에 설정되어진 정보 참고해서 Validation 수행하는 코드 자동 삽입 
        //.....
        return member;
    }
}

```

## 그럼 표준 애노테이션으로 커버가 안될때는 어떻게 해야하나?
 - 재활용 많이 할 거 같으면 : 표준 자바 validation을 커스텀 annotation으로 만들어서 그 검증 로직을 구현하면 된다.
 - 한 두번 쓰고 말거 같으면 : 레코드 생성자를 이용해서 그 안에서 로직 검사
